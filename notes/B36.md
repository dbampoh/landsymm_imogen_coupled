#### 2.2.1 Deep-dive: `Q_NON_CO2` RF-level fallback at lines 3974-3984

Inside `SUBROUTINE RADF_NON_CO2` (lines 3930-4024):

```fortran
DATA YEARS   / 1859,  1875,  1890,  1900,  1917
&,             1935,  1950,  1960,  1970,  1980
&,             1990,  2005,  2020,  2030,  2040
&,             2050,  2060,  2070,  2080,  2090
&,             2100, 279*2100 /

DATA Q_NON_CO2/ 0.0344, 0.0557, 0.0754, 0.0912, 0.1176
&,              0.1483, 0.1831, 0.2387, 0.3480, 0.4987
&,              0.6627, 0.8430, 0.9225, 0.9763, 1.0575
&,              1.1486, 1.2316, 1.3025, 1.3604, 1.4102
&,              1.4602, 279*1.4602 /

IF(FILE_NON_CO2) THEN
  OPEN(72,FILE=FILE_NON_CO2_VALS,STATUS='OLD')
  DO I = 1,NYR_NON_CO2
    READ(72,*) YEARS(I),Q_NON_CO2(I)
  ENDDO
  CLOSE(72)
ENDIF
```

**Critical characterisation**:
- The DATA values are **RADIATIVE FORCING (W/m²; "Q" in the variable name)**, NOT emissions. They represent a historical non-CO2 GHG forcing trajectory consistent with the IPCC IS92A-era reference scenario (per the values matching IS92A non-CO2-GHG-bundled forcing series; CH4 + N2O + halocarbons + tropospheric O3 + stratospheric O3 + tropospheric water vapour from CH4 oxidation, combined).
- **Conditionally overridden** when `FILE_NON_CO2=.TRUE.` (line 3991-3997): engine opens `FILE_NON_CO2_VALS` and READS user-supplied (year, Q) pairs INTO the same `YEARS()` + `Q_NON_CO2()` arrays. The DATA-statement values are clobbered by the file-read on every engine call. So when the v1.0 .ins sets `FILE_NON_CO2=1` (which it does at line 281), the engine uses the file values, not the embedded DATA.
- **NOT consumed in the no-`FILE_NON_CO2_VALS` path that engine would take if FILE_NON_CO2=FALSE**: the DATA-statement values would only be used as a literal fallback if the user explicitly disabled `FILE_NON_CO2`. This is a low-risk fallback path (engineer opt-in).

#### 2.2.2 Verdict on sub-item (a)

✅ **NO emission-rate embedded constants found.** The only RF-level fallback is conditionally overridden in v1.0; the other DATA statements are calendar metadata, filesystem path stems, physical constants of the Joos ocean model, humidity lookup tables, or B33(c) SAVE guards. **B36-IRRELEVANT-by-finding.**

### 2.3 Sub-item (b) — Default-filename string constants

`SUBROUTINE SETTIN` (lines 1635-1879) parses `imogen_settings.txt` to populate the engine's runtime config. Initialisation block at lines 1699-1707:

```fortran

```

All 6 `FILE_*` + 3 `DIR_*` string variables initialise to **single space `' '`** (effectively empty). Then the SELECT-CASE label-matching loop at lines 1717-1861 populates any that appear in `imogen_settings.txt`. After the loop closes, the variables that were NOT explicitly set retain the `' '` initialisation.

**The engine has zero hardcoded default-filename fallback** for any of these. If a user's `.ins` lacks (e.g.) `FILE_NON_CO2_VALS`, the variable stays as `' '` and the eventual `OPEN(72, FILE=FILE_NON_CO2_VALS, STATUS='OLD')` at line 3992 fails with a Fortran I/O error ("file not found"). Engine aborts with a clear error message rather than silently consuming an embedded default.

#### 2.3.1 Verdict on sub-item (b)

✅ **NO default-filename fallback strings found.** Missing FILE_* params cause `OPEN(...)` errors rather than silent fallback to embedded defaults. **B36-IRRELEVANT-by-finding.**

### 2.4 Sub-item (c) — BLOCK DATA subroutines

`grep -iE "^\s*(BLOCK DATA)\s+\S"` across all 4700 LOC returns **ZERO matches**. There are no `BLOCK DATA` subroutines anywhere in the file.

This is the **most important green-flag finding** of the audit. BLOCK DATA is Fortran's mechanism for pre-populating COMMON-block emission arrays before any subroutine fires. Its complete absence in `imogen_lpjg.f` means no such pre-population can occur silently.

#### 2.4.1 Verdict on sub-item (c)

✅ **ZERO `BLOCK DATA` subroutines.** **B36-IRRELEVANT-by-finding.**

## 3. Engine-source-level observability features supporting the no-double-counting invariant

The Fortran engine does NOT silently fall back to embedded defaults (per the sub-item findings above), but it ALSO does NOT hard-assert the **user-data** design invariants that the .ins-layer config must respect to prevent double-counting between FILE_NON_CO2_VALS (RF) + FAIR (CH4+N2O) + LPJG-flux (natural CH4+N2O). Instead, it surfaces these invariants via two inline PRINT warnings at the relevant flow points:

| Line | PRINT warning | Design invariant it documents | Fires when |
|---|---|---|---|
| `imogen_lpjg.f:655` | `'Reading CH4 and N2O fluxes from LPJ-GUESS - ensure FILE_CH4_N2O_EMITS only contains anthropogenic emissions'` | `FILE_CH4_N2O_EMITS` must contain ONLY anthropogenic CH4+N2O (natural fraction comes via `FILE_LPJG_CH4_N2O_FLUX` separately) | LPJG-flux read block, when `NONCO2_EMISSIONS_LPJG=TRUE` (v1.0 sets it 1) |
