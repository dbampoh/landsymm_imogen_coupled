## 1. Background — the path-iv sidecar mechanism (B37 recap)

Per `notes/B37.md` §2.5 + §3.2 (and confirmed at B40 closeout):

**The F-10 case-α problem**: in single-process `imogencfx` mode, `imogencfx::init()` inline-calls `RUN_IMOGEN_ENGINE()` at `lpjguess/modules/imogencfx.cpp:549` which doesn't return until the engine exits. The engine's per-iter polling loop at `lpjguess/modules/climatemodel.cpp:347-401` waits for an LPJG `done` marker that LPJG can't write because LPJG's main loop never runs (it's blocked by the inline engine call). Without intervention, the engine deadlocks after a FIRSTCALL-bypass-permitted "free run" of (`K_flip + 1`) outer iters per the closed-form formula at `notes/B37.md` §2.3.

**B37 path-iv mechanism**: spawn a background bash sidecar that touches `<HSHAKE_DIR>/done` every 1 second. The engine's polling loop (3-second sleep between cycles) mostly sees `done` on each poll; consumes the flux files (read from intermediary-py-derived Option-B static files); deletes `done` at line 578; sidecar re-creates it within 1 sec; engine progresses to the next year. This bypasses the F-10 deadlock without requiring source change OR F-12 architectural fix.

**B37 DR1 empirical confirmation** (2026-05-19 evening): the ad-hoc DR1 harness at `_chat_artifacts/b37_diagnostic_run_2026-05-19/dr1_treatment_with_sidecar.sh` produced 202 year-dirs (1900-2101) in 12 min 48 sec wall on smoke 4-cell config. Engine exits 99 over-shoot at YEAR1==2100 hardcoded boundary (per the brittle sentinels at `climatemodel.cpp:1189-1197`; B45 NEW filed for v1.1+ source-edit cleanup).

**B44 productisation goal**: turn the path-iv mechanism into a first-class launcher flag so:
- Future cluster sbatch wrappers can pass `--engine-only-mode` cleanly (cleaner than duplicating sidecar logic OR path-depending on DR1 harness)
- Future maintainers + operators don't need to remember the bash sidecar incantation
- Reviewer-defensibility for the paper methods section is clean ("via `scripts/run_coupled.sh --engine-only-mode`" vs "via an ad-hoc bash sidecar wrapper")
- Sidecar PID is properly cleaned up via trap-on-EXIT on any exit path (normal/error/signal/timeout)
- Launcher's exit-99 messaging context-switches to "engine-only-mode terminal" narrative (more accurate than the F-10-deadlock-workaround text)

```bash
if [ "${ENGINE_ONLY_MODE}" = "1" ]; then
  case "$COUPLING_MODE" in
    prescribed) ;;
    *) _err "--engine-only-mode requires --coupling-mode prescribed; got '$COUPLING_MODE'."
       _err "  Rationale: --engine-only-mode productises the path-iv launcher-side done-marker sidecar"
       _err "  per B37 DR1. It only makes sense with prescribed mode (the v1.0 paper-publication"
       _err "  default). loose bypasses LPJG handshake at input-module not engine; tight is the"
       _err "  F-10-deadlocked mode the sidecar works around (but tight = prescribed in v1.0 since"
       _err "  LPJG main loop never runs either way). See notes/B44.md + notes/B37.md sec 5."
       exit 2 ;;
  esac

```

Rationale for the validation:
- `loose` bypasses the LPJG handshake at the INPUT-MODULE layer (`lpjguess/modules/imogenoutput.cpp:269-274 + 473-478` early-return when mode=='loose'), NOT at the engine layer. The sidecar would be a no-op for loose mode.
- `tight` is the F-10-deadlocked mode the sidecar is designed to work around. In v1.0, tight + sidecar would produce identical behavior to prescribed + sidecar (because LPJG main loop never runs in either case per `imogencfx::init` inline `RUN_IMOGEN_ENGINE` call). Better to require explicit `prescribed` mode for clarity.
- `prescribed` is the canonical v1.0 production-IMOGEN mode per the v1.0 paper-publication architecture decision (B43; F-12 deferred to v1.1+).

### 2.4 Chunk 4 — sidecar spawn + cleanup trap + context-switched warning (~60 LOC at engine launch site)

```bash
SIDECAR_PID=""
cleanup_sidecar() {
  if [ -n "${SIDECAR_PID}" ]; then
    kill "${SIDECAR_PID}" 2>/dev/null || true
    wait "${SIDECAR_PID}" 2>/dev/null || true

  fi

trap cleanup_sidecar EXIT
```

**Design notes**:
- Bash supports both ERR trap (existing at line 112) AND EXIT trap (B44 addition) simultaneously
- `trap cleanup_sidecar EXIT` ensures the sidecar is killed on every exit path: normal completion, error, signal, timeout
- `kill ... 2>/dev/null || true` is defensive (the sidecar may already be dead)
- The happy-path `cleanup_sidecar` call after the engine launch is for prompt cleanup so step 7 (run summary) runs without the background process touching files; the EXIT trap is the safety net
